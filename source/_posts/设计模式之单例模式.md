## Singleton
属于 对象性能 模式

面向对象 解决了 抽象的问题 但是会不可避免付出一些代价(虚函数) 某些情况下希望减少面向对象所带来的成本

有一些特殊的类 必须保证它在系统中只存在一个实例 的情况下使用

```c++
class Singleton{
// 设置构造函数和拷贝构造函数为私有 不被外界使用
private:
    Singleton();
    Singleton(const Singleton& other);
public:
    static Singleton* getInstance();
    static Singleton* m_instance;
};
Singleton *Singleton::m_instance = nullptr;
```

第一种实现 线程非安全版本

```c++
Singleton *Singleton::getInstance() {
    // 线程调度的时候 有可能出现 对象被创建多次的情况
    if (m_instance == nullptr) {
        m_instance = new Singleton();
    }
    return m_instance;
}
```

第二种实现 线程安全版本 加锁 但是锁的代价太高了

```c++
Singleton *Singleton::getInstance() {
    Lock lock;
    // 线程会在此等很久 读操作不需要加锁 因此可以进行优化
    if (m_instance == nullptr) {
        m_instance = new Singleton();
    }
    return m_instance;
}
```

第三种实现 双检查锁 但是 内存读写 Reorder 不安全 导致双检查锁失效

正常的执行流程应当是
1. 先分配内存
2. 调用构造函数
3. 赋值给 m_instance

但是指令执行序列有可能会被 CPU Reorder
1. 先分配内存
2. 直接赋值给 m_instance
3. 调用构造函数

那么有可能 一个线程 调用 getInstance() 返回的只是指向一块原生内存的指针
```c++
Singleton *Singleton::getInstance() {
    if(m_instance == nullptr){
        Lock lock;
        if (m_instance == nullptr) {
            m_instance = new Singleton();
        }
    }
    return m_instance;
}
```

第四种实现 解决 Reorder 问题

```c++
std::atomic<Singleton*> Singleton::m_instance;
std::mutex Singleton::m_mutex;

Singleton *Singleton::getInstance() {
    Singleton* tmp = m_instance.load(std::memory_order_relaxed);
    // 获取内存fence 类似内存屏障 防止指令被 Reorder
    std::atomic_thread_fence(std::memory_order_acquire);
    if (tmp == nullptr) {
        std::lock_guard<std::mutex> lock(m_mutex);
        tmp = m_instance.load(std::memory_order_relaxed);
        if (tmp == nullptr) {
            tmp = new Singleton;
            // 释放内存fence 防止指令被 Reorder
            std::atomic_thread_fence(std::memory_order_release);
            m_instance.store(tmp, std::memory_order_relaxed);
        }
    }
    return tmp;
}
```
