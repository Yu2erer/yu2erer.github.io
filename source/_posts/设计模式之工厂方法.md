## Factory Method
属于对象创建模式

在软件系统中 经常有创建对象的工作 但是由于需求的变化 需要创建的对象的具体类型会经常变化

定义一个用于创建对象的接口 让子类决定实例化哪个类 Factory Method 使得一个类的实例化延迟(目的是 为了解耦 解的是new xxx这个具体类的耦合)到子类

下面来看一个例子 现在是要在主界面里面加一个 文件分割器 但是文件分割器有很多种类型(二进制 txt 图片 视频....) 而具体是什么分割器 是会随着需求的变化而变化的

```c++
class MainForm : public Form {
    TextBox *txtFilePath;
    TextBox *txtFileNumber;
    ProgressBar *progressBar;
public:
    void Button1_Click() {
        ISplitter *splitter = new BinarySplitter(); // 依赖具体类
        splitter->split();
    }
};
class ISplitter {
public:
    virtual void split() = 0;
    virtual ~ISplitter() { };
};
class BinarySplitter : public ISplitter { };
class TxtSplitter: public ISplitter { };
class PictureSplitter: public ISplitter { };
class VideoSplitter: public ISplitter { };
```

以上 可以看到 我们有一个创建对象的地方 依赖了具体类 问题是现在的需求变化了 需要创建的对象的具体类型跟着变化了 那么我们要么只能回到 `MainFrom` 这个类里面 去改写 `void Button1_Click()` 函数 但这显然不合理啊 因此就有了 下面的工厂方法的例子

```c++
class ISplitter {
public:
    virtual void split() = 0;
    virtual ~ISplitter() { };
};
class BinarySplitter : public ISplitter { };
class TxtSplitter: public ISplitter { };
class PictureSplitter: public ISplitter { };
class VideoSplitter: public ISplitter { };
// 工厂基类
class SplitterFactory {
public:
    virtual ISplitter* CreateSplitter() = 0;
    virtual ~SplitterFactory() { };
};
// 具体的工厂
class BinarySplitterFactory: public SplitterFactory {
public:
    virtual ISplitter* CreateSplitter() {
        return new BinarySplitter();
    }
};
class TxtSplitterFactory: public SplitterFactory {
public:
    virtual ISplitter* CreateSplitter(){
        return new TxtSplitter();
    }
};
// .... 剩下的几个工厂类就不写了
class MainForm : public Form {
    SplitterFactory *factory;
public:
    // 可以是不同的工厂 然后不同的工厂能够创建出不同的对象
    MainForm(SplitterFactory *factory) {
        // 此处的 factory 会产生依赖问题
        // 比如 你未来要实现某个 factory 类 去创建某个对象
        // 但是 MainForm 里面已经不再依赖具体类了
        // 至于外面的依赖 不归我们管
        this->factory = factory;
    }
    void Button1_Click() {
        ISplitter *splitter = factory->CreateSplitter(); // 多态new
        splitter->split();
    }
};
```

以上我们能够看出 我们因为要创建对象 但是 对象可能会多变 如果直接改源码 会非常的傻 此时我们使用工厂方法 不同的工厂创建不同的对象 而具体是什么工厂 则是由外界传进来 这样 工厂是多态的 工厂创建的对象也是多态的 但是同时还是会产生依赖问题 具体探讨可以看上面代码注释

Facotry Method模式 用于隔离类对象的使用者和具体类型之间的耦合关系 面对一个经常变化的具体类型 紧耦合关系(new 对象)会导致软件的脆弱

缺点是要求创建对象的方法/参数相同
