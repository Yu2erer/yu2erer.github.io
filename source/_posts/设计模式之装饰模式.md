## Decorator

装饰模式最核心的思想就是 用组合代替继承

继承不一定是最好的 当 过度地使用继承来扩展对象的功能 会导致 子类急剧增多 缺乏灵活性 之所以会过度地使用继承是因为需求快速变化

动态(组合)地给一个对象增加一些额外的职责 就增加功能而言 Decorator模式比生成子类(继承)更为灵活(减少代码冗余 减少子类个数)

假设现在要写一个流的功能 有很多个流 自然而然就会想将其抽象出来

一开始的版本
```c++
class Stream {
public：
    virtual char Read(int number) = 0;
    virtual void Seek(int position) = 0;
    virtual void Write(char data) = 0;
    virtual ~Stream(){}
};
class FileStream : public Stream {
public:
    virtual char Read(int number) { /* 读xxx */ }
    virtual void Seek(int position) { /* 定位xxx */ }
    virtual void Write(char data) { /* 写xxx */ }
};
class NetworkStream :public Stream{
public:
    virtual char Read(int number) { /* 读xxx */ }
    virtual void Seek(int position) { /* 定位xxx */ }
    virtual void Write(char data) { /* 写xxx */ }
};
// 此时有一个新的需求 文件流和网络流要进行加密操作
class CryptoFileStream : public FileStream {
public:
    virtual char Read(int number){
        // 额外的加密操作...
        FileStream::Read(number);
    }
    virtual void Seek(int position){
        // 额外的加密操作...
        FileStream::Seek(position);
        // 额外的加密操作...
    }
    virtual void Write(byte data){
        // 额外的加密操作...
        FileStream::Write(data);
        // 额外的加密操作...
    }
};
// 注意这个类和上面这个类 你能发现很多地方的代码重复了
// 都是 XXXStream::XXX 重复性高
class CryptoNetworkStream : :public NetworkStream{
public:
    virtual char Read(int number){
        // 额外的加密操作...
        NetworkStream::Read(number);
    }
    virtual void Seek(int position){
        // 额外的加密操作...
        NetworkStream::Seek(position);
        // 额外的加密操作...
    }
    virtual void Write(byte data){
        // 额外的加密操作...
        NetworkStream::Write(data);
        // 额外的加密操作...
    }
};
// 又有一个新的需求 对流进行缓冲 又要像上面一样做
// 再来一个新的需求 对流既缓冲又加密 又要再继承一次
// 可以看出代码有冗余 各种重复
class CryptoBufferedFileStream : public FileStream {
public:
    virtual char Read(int number){
        // 额外的加密操作...
        // 额外的缓冲操作...
        FileStream::Read(number); // 代码冗余
    }
    virtual void Seek(int position){
        // 额外的加密操作...
        // 额外的缓冲操作...
        FileStream::Seek(position);
        // 额外的加密操作...
        // 额外的缓冲操作...
    }
    virtual void Write(byte data){
        // 额外的加密操作...
        // 额外的缓冲操作...
        FileStream::Write(data);
        // 额外的加密操作...
        // 额外的缓冲操作...
    }
};
```

一下子子类的数量就增加了 还出现了不少冗余代码 出现冗余代码的原因是 错误的使用了继承

修改后的版本
```c++
class Stream {
public：
    virtual char Read(int number) = 0;
    virtual void Seek(int position) = 0;
    virtual void Write(char data) = 0;
    virtual ~Stream(){}
};
class FileStream : public Stream {
public:
    virtual char Read(int number) { /* 读xxx */ }
    virtual void Seek(int position) { /* 定位xxx */ }
    virtual void Write(char data) { /* 写xxx */ }
};
class NetworkStream :public Stream{
public:
    virtual char Read(int number) { /* 读xxx */ }
    virtual void Seek(int position) { /* 定位xxx */ }
    virtual void Write(char data) { /* 写xxx */ }
};
// 可以看到以上三个类不变 以下扩展一个中间类 用来给扩展出来的功能做基类
class DecoratorStream : public Stream {
protected:
    Stream* stream;
    DecoratorStream(Stream *stm) : stream(stm) { }
};
// 可以看到使用这种组合 + 多态的方式 能够快速的为 File 和 Network
// 增加一个加密的功能 子类数量减少
class CryptoStream : public DecoratorStream {
public:
    CryptoStream(Stream *stm) : DecoratorStream(stm) { }
    virtual char Read(int number) {
        // 额外的加密操作...
        stream->Read(number);
    }
    virtual void Seek(int position){
        // 额外的加密操作...
        stream::Seek(position);
        // 额外的加密操作...
    }
    virtual void Write(byte data){
        // 额外的加密操作...
        stream::Write(data);
        // 额外的加密操作...
    }
};
// 同样 采用这种方式快速为 File 和 Network 增加缓冲的功能
class BufferedStream : public DecoratorStream{
public:
    BufferedStream(Stream *stm) : DecoratorStream(stm) { }
    // ......
};
```

综上 我们能看出 使用了装饰模式 可以大幅度的减少子类的数量和代码冗余 还知道 继承在特定情况下 并不如组合好用 不应该盲目使用继承

特别要说明的是 Decorator模式并非是用来解决多子类衍生的多继承问题 而是应用于 主体类在多个方向上的扩展功能 这就是(装饰)模式的名字来源
