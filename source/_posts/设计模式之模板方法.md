### Template Method
可分为 行为型模式 或者 组件协作模式 

定义一个操作中的算法的骨架(稳定) 而将一些步骤延迟(变化) 到 子类中 Template Method使得子类可以不改变(复用) 一个算法的结构即可重定义(override)该算法的某些特定步骤

用于在确定稳定操作结构的前提下 灵活应对各个子步骤的变化

现在来看例子 按照上面的设定 在重构之前 就是下面的样子

<!-- more -->

```c++
// 程序库
class Library {
public:
    void Step1() {}
    void Step3() {}
    void Step5() {}
};
// 应用程序库
class Application {
public:
    bool Step2() {}
    void Step4() {}
};
int main() {
    Library lib();
    Application app();

    lib.Step1();

    if (app.Step2()) {
        lib.Step3();
    }

    for (int i = 0; i < 4 ; i ++) {
        app.Step4();
    }

    lib.Step5();
}
```

这上面可以看出 框架的开发人员完全可以把这整个流程实现好 然后让 应用程序库的开发人员 去重写 Step2() 和 Step4()

```c++
// 程序库
class Library {
public:
    // 稳定的 Template Method
    void Run() {
        Step1();

        if (Step2()) {
            Step3(); // 虚函数多态绑定
        }

        for (int i = 0; i < 4; i ++) {
            Step4(); // 虚函数多态绑定
        }

        Step5();
    }
    virtual ~Library(){ }
protected:
    void Step1() {}
    void Step3() {}
    void Step5() {}
    // 推荐声明为 protected 因为这两个函数在被子类重写之前 被外界单独调用是毫无意义的
    virtual bool Step2() = 0;
    virtual void Step4() = 0;
};
// 应用程序库
class Application : public Library {
protected:
    virtual bool Step2() { // 子类在此重写 }
    virtual void Step4() { // 子类在此重写 }
};
int main() {
    Library *pLib = new Application(); // 多态 动态绑定
    pLib->Run();

    delete pLib;
}
```

Library 实现的早 Application 实现的晚 晚的去调用早的 就叫早绑定
但是面向对象更好的实现是 Library 早 去调用 晚实现的 Application

晚绑定好过早绑定 改良后应用程序的开发人员负担能减轻

Template Method 更应该理解为样板方法 使得子类的开发者可以直接复用 但是又能支持子类的变化
